How it works: BYOK architecture

Automaite is built on a Bring Your Own Keys model. The AI agents — Claude Code, Gemini CLI, Codex — run directly on your own computer via the desktop app. Automaite's servers do not run AI models on your behalf, and your code, files, and prompts never pass through our infrastructure.

Our server acts as a pure relay: it opens an authenticated WebSocket tunnel between your browser or phone and your own computer. When you type a command on your phone, it travels over TLS to our relay, then over another TLS connection to your desktop. The relay reads just enough to route the connection — it does not inspect or store what passes through it.

This design means that even if Automaite's servers were compromised, an attacker would find nothing useful: no AI model outputs, no terminal history, no plaintext credentials.

🔑

Your API keys

To launch an agent, Automaite needs to inject your Anthropic, Google, or OpenAI API key as an environment variable on your computer. You provide these keys once; from that point on the desktop app handles the injection automatically.

Encryption at rest

Keys are stored encrypted using Fernet (AES-128-CBC with PKCS7 padding, authenticated with HMAC-SHA256). The encryption key is derived from your JWT session secret — a secret that is never written to disk on our servers.

Fernet(key).encrypt(api_key_bytes) → stored ciphertext only

At session start

When you launch an agent session, the ciphertext is decrypted in memory, injected as an environment variable into the agent process, and then immediately discarded from memory. Keys are never written to logs, never echoed to the terminal, and never transmitted over the WebSocket relay.

What we store

We store the encrypted ciphertext and a masked display name (e.g. "sk-ant-...xxxx") so you can identify which key is saved. We do not store the plaintext key, and our servers cannot reconstruct it without the JWT secret that is scoped to your session.

🛡️

Credential vault

The credential vault lets you store SSH keys, passwords, and secrets on your phone, gated behind biometric authentication (fingerprint or face unlock). This is entirely optional — you can use Automaite without it.

How a credential request works

  1. A running AI agent decides it needs a credential (e.g. an SSH password to deploy code).
  2. It sends a structured request to the relay, which forwards it to your phone.
  3. Your phone displays the request: agent name, credential name, and the reason the agent gave.
  4. You approve with your fingerprint or face. The credential is encrypted on-device using AES-256-GCM with an HKDF-derived key tied to the current session.
  5. The encrypted blob travels through the relay to your desktop, where the agent process decrypts it locally.

Zero-knowledge relay

The relay routes the encrypted blob from your phone to your computer — it never holds the decryption key and never decrypts the credential. Even in a worst-case scenario where the relay server is fully compromised, vault credentials remain protected because the key never leaves your devices.

relay sees: encrypted_blob (AES-256-GCM) → cannot decrypt

Biometric gating (Android)

On Android, credentials are stored using the Android Keystore system — a hardware-backed secure enclave. The decryption key is bound to biometric authentication and cannot be extracted from the device, even by a rooted OS.

🪪

Authentication

Automaite uses Google OAuth 2.0 only — there are no passwords to store, no password reset flows, and no password database to leak. Your Google account is the sole authentication factor.

  • JWT session tokens are issued after a successful OAuth exchange and stored in HttpOnly, Secure cookies — inaccessible to JavaScript on the page.
  • Nonce verification on Google's ID token prevents replay attacks (each token can only be used once).
  • Rate limiting on login: 5 failed attempts triggers a 24-hour block on that account to prevent credential stuffing.
  • WebSocket session ownership: each WebSocket connection is validated against the authenticated user's session — you can only connect to your own agent sessions, never someone else's.
📡

Data in transit

All communication between your browser, phone, relay server, and desktop app travels over TLS (HTTPS and WSS). The relay is served by Caddy with automatic Let's Encrypt certificate renewal — no manual certificate management and no chance of an expired cert quietly downgrading to plaintext.

  • WebSocket rate limiting: each connection is capped at 100 messages per second to prevent abuse.
  • Message size limit: maximum 64 KB per WebSocket frame.
  • Origin checking: WebSocket upgrade requests are validated against allowed origins — connections from unexpected origins are rejected before the tunnel is established.
🧱

Multi-tenancy isolation

Automaite is a multi-user service, and session isolation is enforced at the server level:

  • Agent connections are registered per user email — there is no shared agent pool where one user's session could spill into another's.
  • Session lookup is gated by ownership: a request to attach to a session is rejected unless the authenticated user owns that session.
  • Users cannot enumerate, view, or interact with sessions belonging to other accounts.

In short: your sessions are yours. No other Automaite user can reach them.

🔍

What Automaite can and cannot see

We believe you should know exactly what visibility we have into your activity. Here it is, plainly:

Your Google email
Yes — received from OAuth, used to identify your account
API key contents
No — stored as ciphertext only; we cannot read the plaintext
Masked key name
Yes — e.g. "sk-ant-...xxxx" so you can identify saved keys
Session metadata
Limited — start time, agent type, duration; not session content
Terminal output
No — buffered in-memory only (256 KB scrollback); not written to disk or logs; lost on relay restart
Vault credentials
No — zero-knowledge relay; we route the encrypted blob, never the plaintext
Your code / files
No — agents run on your computer; file contents never reach our servers
AI model outputs
No — model calls go directly from your computer to the AI provider (Anthropic, Google, OpenAI)
📖

Open source (planned)

We intend to release the full server and client code — including the relay, the desktop agent connector, and the crypto layer — on GitHub under an open source licence. Community members will be able to audit:

  • The relay code: how connections are routed and what (if anything) is buffered
  • The agent connector: how API keys are injected and how vault requests are handled
  • The crypto code: the Fernet key derivation, the AES-256-GCM vault encryption, and the HKDF key schedule

We are not open source yet. Until then, this document is our commitment to transparency. If you have questions that go beyond what is written here, please reach out.
